    CIF Design Library

    Show Grid

    This design library is a collection of design elements and modules to be used throughout CIF's online properties. The design library also serves as documentation for how to use these modules as well. Whenever a new design module is created, it is highly encouraged that it be documented here so that future maintainers can quickly learn how to implement the module.

    The maintainers of CIF's web properties are encouraged to learn how to curate this library.

    Elements may not be perfectly aligned to the grid system in some browsers because relative units are used for margins and font sizing. Using relative units ensures that our design is scalable, but some browsers may round these values differently and yield slightly different pixel sizes.

    There may also be an uneven application of the background color on some elements. This is not a bug; many elements on this page are placed on differently colored backgrounds to accentuate them, while on an actual page they would not be.

    Header

    The header is a major navigational landmark for the design. It should contain the CIF logo (which should link to the homepage) and the site's primary navigation. Only a single header can exist on a page.

    Headers are created by giving an element a header id. The header's contents should be contained within an element with a content class. If an element has a header-logo class, that element's contents will be replaced with the CIF logo. The element with the header-logo class should contain an <a> element, which preferably links to the home page.

    Any menus in the header will have additional styles applied to them.

    Sub-Headers

    Sub-headers are intended for use as headers for major sections of content. They should be used sparingly.

    Because sub-headers span the width of the page, they can not be contained within an element with a content class. Instead, they must be used outside the content wrapper.

    Sub-headers themselves are created by applying a sub-header class to an element. The contents of a sub-header should be wrapped in an element with a content class. Textual contents may be wrapped with a sub-header-body class to better align them with modules such as right-aligned menus.

    Grid Layout

    Grid layouts can be constructed manually by creating a wrapper element for the row with a *-columns class, where * specifies the number of columns (either two or four). The row should contain 2 to 4 elements with a column-# class, where # is the number of the column (1, 2, 3, etc).

    A column class can also be used for columns, where the last column has a an additional last class.

    Manually specifying a grid layout should be avoided if possible. These classes are primarily intended for defining two-column layouts. If you need finer control over the grid, consider creating a new class in the sass/modules directory which has the layout you need. The more generalized your module is, the more reusable it will be throughout the design.

    Fluid Rows

    A fluid row is only as wide as its contents. A row becomes fluid by adding a fluid class to the row container.

    Please Note

    Fluid grids are floated left, so their containers may collapse to have 0 height if there is not a non-floated sibling element. You may need a container which has a clearfix to contain the fluid grid.

    Forms

    Form elements do not require special classes for their styling, so proper markup is necessary to prevent form elements from becoming unruly. All form inputs should be wrapped in a <label> with an appropriate text label for the field. Multiple related fields should be contained in a <fieldset>, with a <legend> if appropriate. See the markup of the form below for a better idea of how forms should be marked up.

    Image Captions

    Image and figure captions can be created by using the <figcaption> element or applying a caption class to an element. If using <figcaption>, please be aware of its semantics and the semantics of the <figure> element.

    It should also be kept in mind that images will not always line up with the vertical rhythm of the page, so captions and anything under the image may be offset from the page's vertical rhythm.

    Show Markup

    Because it is impossible to set the margin under an image to fill in the gap it creates with the vertical rhythm using pure CSS, captions may be out of sync with the vertical rhythm of the page. While this isn't a big deal, the only way to deal with it is to use JavaScript to set the margins under only those images which should maintain the vertical rhythm of the page (gross, I know), or just use images in columns next to text columns which extend past the image (like I'm doing here to keep this page's vertical rhythm together).

    Code Blocks

    Code blocks are created by placing code within a <code> tag nested within a <pre>. Syntax highlighting is performed with JavaScript via Prism. To activate highlighting for a block of code, it must have a language-* class, where * is the name of the language (markup, css, c, etc).

    Please keep whitespace in mind when marking up your code. The <pre> tag will pick up all newlines, so there should be no whitespace between the first and last line of your code and the opening and closing tags.

    Icons

    Icons can be added to text by applying an icon-* class, where * is a supported icon. These classes should only be applied to inline elements.

    By default, the icon appears before the text it is associated with. It can be placed after the text by adding an icon-after class to the element.

    Text can be replaced by an icon by wrapping the text in a screen-reader-text class. This removes the text from the page, while still leaving it accessible to screen readers and robots.

    Block Alignment

    Content blocks can be aligned with the align-left and align-right classes.

    Please Note

    Use of these classes is highly discouraged because floated elements can cause problems with other elements on the page. Most notably, a container will collapse to have a height of 0 if all of its children are floated.

    The solution to this issue is to apply a clearfix. However, there is no markup class defined for applying a clearfix to discourage the excessive use of these classes. If you need a complex layout, please consider using one of the UI modules defined in this file or create your own module in the sass/modules directory.

    Hero Block

    A hero block is intended to be the first thing a user sees when visiting the site. It consists of a hero graphic and a hero body, which should contain introductory text.

    The hero block should be contained within an element with a hero class. The <img> element should be wrapped in an element with a hero-graphic class, and the hero body should be contained within an element with a hero-body class.

    Articles

    An article is a block of serialized content with an article class. All articles should be contained inside an element with an articles class. Because serialized content usually has metadata associated with it, applying an article-metadata class to the metadata's container will style its contents appropriately.

    Articles were specifically designed with the dimensions of a sidebar in mind. It is highly encouraged to pair a collection of articles with a sidebar containing archive navigation.

    The example code uses <article> tags to wrap each article, and <footer> tags to wrap metadata. This is actually the semantically correct usage of <footer>, see the docs.

    Profile

    A profile is a short block of content describing something specific, optionally with a profile image. A profile is contained within a wrapper with a profile class. Profile images should be wrapped within an element with a profile-image class — the <img> element itself should not have this class. The body of the profile should be wrapped in an element with a profile-body class. The body contents can be whatever you need.

    Footer

    The footer is a major navigational landmark for the design. This is where copyright information and random quotes fit best. Secondary navigation also belongs in the footer.

    Footers are created by giving an element a footer id. The footer's contents should be contained within an element with a content class.

    Any menus in the footer will have additional styles applied to them. Elements with a copyright or quote class will also be given additional styling.
