    Because the CIF design consists of numerous modules, SASS and Compass were chosen to make development faster and more organized. However, it is understandable that not everyone is familiar with these and that a brief introduction is necessary to get you on your way.

    SASS is an unofficial extension of CSS with two syntaxes; we use the more popular “Sassy CSS” syntax which has an scss file extension. SASS compiles into CSS, making it ideal for organizing code into multiple files which can then be compiled into a single CSS file. Compass is a framework built on top of SASS that provides useful utilities for grid systems, typographic rhythm, and cross-browser CSS3 vendor prefixing. This guide will go over the parts of SASS and Compass that are commonly used in our files.

    Compiling everything into a single stylesheet is much more efficient than making multiple HTTP requests to the server for multiple stylesheet files. It is worth noting that the @import directive in SASS does not function as in CSS; the imported file's contents will be injected into the compiled CSS, resulting in a single file. Using @import in pure CSS to request additional stylesheets is highly inefficient and results in additional HTTP requests.

    You can read more about SASS's @import at the official documentation.

    How SASS Handles Files

    SASS files have an scss extension and will be compiled into a corresponding CSS file. However, SASS files prefixed with an underscore are considered “partials” and will not be compiled into their own CSS file. Because we want to compile our styles into a single stylesheet, most of our SASS files are partials which are imported into a master file (style.scss in our case). The master file is then the only file compiled into CSS, leaving us with a single stylesheet containing all of our code.

    When importing a partial in SASS, the underscore can be left off of the filename.

    _my-module.scss
    .my-module { color: #f0f; }
    style.scss
    body { color: #222; }
@import 'my-module';

    Compiles into…

    style.css
    body { color: #222; }
.my-module { color: #f0f; }

    SASS Syntax

    As an extension of CSS, SASS's syntax is not much different. In fact, pure CSS is also valid SASS, so if you feel most comfortable writing CSS then you can do so, although we recommend at least using some of SASS's more basic features.

    • Variables

      SASS provides the ability to define variables in your stylesheets. This is useful for common values such as colors, font stacks, or grid dimensions. Please note that these values will be compiled into hardcoded values in the final CSS, as CSS does not natively support variables.

      For more information on SASS variables, see the official documentation.

      $serif-typeface: 'Adobe Garamond Pro', Georgia, serif;
$h1-font-size:   48px;
$text-color:     #222;

      Variable Scope

      SASS variables are prefixed with a $ and are assigned much like CSS properties. SASS variables also have scope. A variable defined outside any declaration will be available everywhere, while those defined within a declaration are local to that declaration.

      $my-color: red;

.red { color: $my-color; /* red */ }

.blue {
	$my-color: blue;
	color: $my-color; /* blue */
}

      Variable Operations

      SASS variables also support mathematical operations.

      For more information on SASS variables operations, see the official documentation.

      $column-width: 90px;
$gutter-width: 30px;
$columns: 8;

$grid-width: ($columns * $column-width) + ($gutter-width * ($columns - 1));

    • Nested Selectors

      SASS allows declarations to be nested. This makes it easier to write and organize complex selectors.

      For more information on nesting selectors, see the official documentation.

      SASS
      p {
	color: grey;
	a { color: blue; }
	em, strong { color: black; }
}
      CSS
      p { color: grey; }

p a { color: blue; }

p em, p strong { color: black; }

      The & Operator

      The & operator is used in SASS to reference the parent selector. This does not allow you to style an element based on its parent element, which is impossible in CSS. This merely allows you to reference the parent selector in a nested selector.

      For more information on referencing parent selectors, see the official documentation.

      SASS
      p {
	color: black;
	&.light-color { color: grey; }
	.large-text & { font-size: 3em; }
}
      CSS
      p { color: black; }

p.light-color { color: grey; }

.large-text p { font-size: 3em; }

    • @extend and Inheritance

      Selectors in SASS can be inherited from via the @extend directive to reduce the duplication of code. This applies the styles of the extended selector to the selector that called @extend.

      For more information on the @extend directive, see the official documentation.

      SASS
      p { margin-bottom: 1.5em; }
ul, ol {
	@extend p;
	list-style-position: outside;
}
      CSS
      p, ul, ol { margin-bottom: 1.5em; }

ul, ol {
	list-style-position: outside;
}

    • Mixins

      Mixins are blocks of styles which can accept arguments. A mixin is defined with the @mixin directive and can accept parameters.

      Common uses for mixins are for vendor prefixing (calling a box-shadow mixin instead of writing out -webkit-box-shadow, -moz-box-shadow, etc) or storing common blocks of code (clearfix code, high resolution background image code, etc).

      @mixin my-mixin($color, $bg: white, $use-border: false) {
	color: $color;
	background-color: $bg;
	
	@if $use-border {
		border: 1px solid $color;
	}
}

      Calling Mixins

      Mixins are called using the @include directive. SASS supports named variables, so values can be passed to specific parameters.

      p { @include my-mixin(black); }
.inverted { @include my-mixin(white, black); }
.border { @include my-mixin(black, $use-border: true); }

    • Control Directives

      Control directives allow styling to be applied based on certain conditions. This is primarily intended for mixins and styles which make use of variables. It is recommended to avoid these directives if possible because they can add bloat and make code more complex. In most cases you shouldn't need anything more than the @if directive.

      For more information on the different control directives, see the official documentation.

      $module-width: 300px;

@if $module-width < 300px {
	font-size: 0.75em;
} @else if $module-width > 1000px {
	font-size: 1.5em;
}

    • Comments

      SASS supports 3 types of comments: multiline CSS comments (/* */), single-line comments (//), and loud comments (/*! */).

      Multiline CSS comments are preserved in the compiled CSS, unless compression is on in which case they are removed (we compress our CSS). Single-line comments are always removed from the compiled CSS and are only visible in the SASS source code. Loud comments are similar to multiline comments, but they are never removed from the compiled CSS. Because of this, loud comments should rarely be used.

      We use a single loud comment at the top of our stylesheet for WordPress, which reads theme metadata from that comment. Loud comments are best used for functional comments like that, which serve a purpose other than style documentation. Multiline and single-line comments should primarily be used for documenting styles to make maintenance easier for future maintainers.

    Using Compass

    Compass is a framework for SASS which provides useful mixins and functions. Compass also makes referencing external files from SASS easier. The config.rb file, which defines the Compass project directory, allows you to change the CSS output style and location of external resources.

    For more information on the config.rb file, see the official documentation.

    Resource Locations

    Compass resources can be located relative to the Compass project directory (the directory with the config.rb file). By default, images, SASS, and CSS are located in their own subdirectories of the Compass project. Fonts for embedding in CSS are located in a fonts directory of the CSS directory by default.

    /compass_project
    /compass_project/sass
    /compass_project/css
    /compass_project/css/fonts
    /compass_project/images

    Compiling SASS

    Because we're using Compass, the command line utility for Compass, not SASS, should be used to compile the stylesheets. Some apps exist to perform this task for you, but the command line is really not bad at all.

    Simply run compass compile [path/to/project], where the path to the project is the path to the directory with the config.rb file.

    $ compass compile ~/Documents/CIF/design/

    For more information on the Compass command line interface, see the official documentation.

    Automatic Compilation

    Alternatively, you can run compass watch [path/to/project] to recompile the stylesheet when any changes to a SASS file are made. This is great for development, because the CSS file will be up to date with your SASS styles within a second. When you're done, you can stop the Compass process by pressing ctrl-c. The only downside to this command is that if changes are made to the config.rb file, you may need to stop the current process, re-run compass watch and save a SASS file to trigger a recompilation. This behavior may change in future versions of Compass.

    $ compass watch ~/Documents/CIF/design/
>>> Compass is polling for changes. Press Ctrl-C to Stop.
>>> Change detected at 04:48:35 to: style.scss
overwrite css/style.css
^C
$
